# Everything Claude Code (ECC) — 에이전트 지침

이것은 소프트웨어 개발을 위한 13개의 전문 에이전트, 50개 이상의 스킬, 33개의 명령어, 그리고 자동화된 훅 워크플로우를 제공하는 **프로덕션 수준의 AI 코딩 플러그인**입니다.

## 핵심 원칙

1. **에이전트 우선** — 도메인 작업은 전문 에이전트에게 위임
2. **테스트 주도** — 구현 전 테스트 작성, 80% 이상의 커버리지 필수
3. **보안 우선** — 보안 타협 금지; 모든 입력값 검증
4. **불변성** — 항상 새 객체를 생성하고, 기존 객체를 직접 수정하지 않음
5. **실행 전 계획** — 복잡한 기능은 코드 작성 전에 계획 수립

## 사용 가능한 에이전트

| 에이전트 | 목적 | 사용 시점 |
|-------|---------|-------------|
| planner | 구현 계획 수립 | 복잡한 기능, 리팩토링 |
| architect | 시스템 설계 및 확장성 | 아키텍처 결정 |
| tdd-guide | 테스트 주도 개발 | 새 기능, 버그 수정 |
| code-reviewer | 코드 품질 및 유지보수성 | 코드 작성/수정 후 |
| security-reviewer | 취약점 탐지 | 커밋 전, 보안 민감 코드 |
| build-error-resolver | 빌드/타입 오류 수정 | 빌드 실패 시 |
| e2e-runner | Playwright 엔드투엔드 테스팅 | 핵심 사용자 흐름 |
| refactor-cleaner | 죽은 코드 정리 | 코드 유지보수 |
| doc-updater | 문서 및 코드맵 업데이트 | 문서 업데이트 시 |
| go-reviewer | Go 코드 리뷰 | Go 프로젝트 |
| go-build-resolver | Go 빌드 오류 수정 | Go 빌드 실패 시 |
| database-reviewer | PostgreSQL/Supabase 전문가 | 스키마 설계, 쿼리 최적화 |
| python-reviewer | Python 코드 리뷰 | Python 프로젝트 |

## 에이전트 오케스트레이션

사용자 프롬프트 없이도 에이전트를 능동적으로 활용:
- 복잡한 기능 요청 → **planner**
- 방금 작성/수정된 코드 → **code-reviewer**
- 버그 수정 또는 새 기능 → **tdd-guide**
- 아키텍처 결정 → **architect**
- 보안에 민감한 코드 → **security-reviewer**

독립적인 작업은 병렬 실행 — 여러 에이전트를 동시에 구동.

## 보안 지침

**모든 커밋 전:**
- 하드코딩된 시크릿 없음 (API 키, 비밀번호, 토큰)
- 모든 사용자 입력 검증
- SQL 인젝션 방지 (파라미터화된 쿼리)
- XSS 방지 (HTML 새니타이즈)
- CSRF 보호 활성화
- 인증/인가 확인
- 모든 엔드포인트에 레이트 리밋
- 오류 메시지에 민감 데이터 노출 금지

**시크릿 관리:** 시크릿 하드코딩 절대 금지. 환경 변수 또는 시크릿 매니저 사용. 시작 시 필수 시크릿 검증. 노출된 시크릿은 즉시 교체.

**보안 이슈 발견 시:** 중단 → security-reviewer 에이전트 사용 → CRITICAL 이슈 수정 → 노출된 시크릿 교체 → 유사 이슈 코드베이스 전체 검토.

## 코딩 스타일

**불변성 (필수):** 항상 새 객체를 생성하고 직접 수정하지 않음. 변경 사항이 적용된 새 복사본 반환.

**파일 구성:** 큰 파일 몇 개보다 작은 파일 다수. 일반적으로 200~400줄, 최대 800줄. 타입이 아닌 기능/도메인별로 구성. 높은 응집도, 낮은 결합도.

**오류 처리:** 모든 레벨에서 오류 처리. UI 코드에서 사용자 친화적인 메시지 제공. 서버 측에서 상세 컨텍스트 로그. 오류를 조용히 삼키지 않음.

**입력 검증:** 시스템 경계에서 모든 사용자 입력 검증. 스키마 기반 검증 사용. 명확한 메시지와 함께 빠른 실패. 외부 데이터를 절대 신뢰하지 않음.

**코드 품질 체크리스트:**
- 함수는 작게 (50줄 미만), 파일은 집중적으로 (800줄 미만)
- 깊은 중첩 없음 (4레벨 초과)
- 적절한 오류 처리, 하드코딩된 값 없음
- 읽기 쉽고 명확한 식별자 이름

## 테스트 요구사항

**최소 커버리지: 80%**

테스트 유형 (모두 필수):
1. **단위 테스트** — 개별 함수, 유틸리티, 컴포넌트
2. **통합 테스트** — API 엔드포인트, 데이터베이스 작업
3. **E2E 테스트** — 핵심 사용자 흐름

**TDD 워크플로우 (필수):**
1. 테스트 먼저 작성 (RED) — 테스트는 반드시 실패해야 함
2. 최소한의 구현 작성 (GREEN) — 테스트가 통과해야 함
3. 리팩토링 (IMPROVE) — 커버리지 80% 이상 확인

실패 트러블슈팅: 테스트 격리 확인 → 목(mock) 검증 → 구현 수정 (테스트가 틀린 경우가 아니면 테스트는 수정하지 않음).

## 개발 워크플로우

1. **계획** — planner 에이전트 사용, 의존성과 리스크 파악, 단계별 분리
2. **TDD** — tdd-guide 에이전트 사용, 테스트 먼저 작성, 구현, 리팩토링
3. **리뷰** — code-reviewer 에이전트 즉시 사용, CRITICAL/HIGH 이슈 처리
4. **커밋** — Conventional Commits 형식, 포괄적인 PR 요약

## Git 워크플로우

**커밋 형식:** `<타입>: <설명>` — 타입: feat, fix, refactor, docs, test, chore, perf, ci

**PR 워크플로우:** 전체 커밋 이력 분석 → 포괄적인 요약 작성 → 테스트 계획 포함 → `-u` 플래그로 푸시.

## 아키텍처 패턴

**API 응답 형식:** 성공 여부 표시, 데이터 페이로드, 오류 메시지, 페이지네이션 메타데이터를 포함하는 일관된 엔벨로프.

**리포지토리 패턴:** 표준 인터페이스(findAll, findById, create, update, delete) 뒤에 데이터 접근을 캡슐화. 비즈니스 로직은 스토리지 메커니즘이 아닌 추상 인터페이스에 의존.

**스켈레톤 프로젝트:** 검증된 템플릿 검색, 병렬 에이전트(보안, 확장성, 연관성)로 평가, 최적의 것을 복제, 검증된 구조 내에서 반복.

## 성능

**컨텍스트 관리:** 대규모 리팩토링 및 멀티 파일 기능에서는 컨텍스트 윈도우의 마지막 20% 사용 자제. 낮은 민감도 작업(단일 편집, 문서, 단순 수정)은 높은 활용도를 허용.

**빌드 트러블슈팅:** build-error-resolver 에이전트 사용 → 오류 분석 → 점진적 수정 → 각 수정 후 확인.

## 프로젝트 구조

```
agents/          — 13개의 전문 서브에이전트
skills/          — 50개 이상의 워크플로우 스킬 및 도메인 지식
commands/        — 33개의 슬래시 명령어
hooks/           — 트리거 기반 자동화
rules/           — 항상 따라야 할 지침 (공통 + 언어별)
scripts/         — 크로스 플랫폼 Node.js 유틸리티
mcp-configs/     — 14개의 MCP 서버 설정
tests/           — 테스트 스위트
```

## 성공 지표

- 모든 테스트 80% 이상의 커버리지로 통과
- 보안 취약점 없음
- 코드가 읽기 쉽고 유지보수 가능
- 성능이 허용 가능한 수준
- 사용자 요구사항 충족
