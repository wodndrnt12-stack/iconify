# 코딩 스타일

## 불변성 (중요)

항상 새 객체를 생성하고, 기존 객체를 직접 수정하지 않는다:

```
// 의사코드
잘못된 예:  modify(original, field, value) → 원본을 직접 변경
올바른 예: update(original, field, value) → 변경사항이 반영된 새 복사본 반환
```

근거: 불변 데이터는 숨겨진 부수 효과를 방지하고, 디버깅을 용이하게 하며, 안전한 동시성을 가능하게 한다.

## 파일 구성

많은 작은 파일 > 적은 대형 파일:
- 높은 응집도, 낮은 결합도
- 일반적으로 200~400줄, 최대 800줄
- 대형 모듈에서 유틸리티 추출
- 타입이 아닌 기능/도메인별로 구성

## 오류 처리

항상 오류를 포괄적으로 처리:
- 모든 레벨에서 명시적으로 오류 처리
- UI 대면 코드에서 사용자 친화적인 오류 메시지 제공
- 서버 측에서 상세한 오류 컨텍스트 로깅
- 오류를 조용히 삼키지 않는다

## 입력 유효성 검사

항상 시스템 경계에서 유효성 검사:
- 처리 전 모든 사용자 입력 유효성 검사
- 가능한 경우 스키마 기반 유효성 검사 사용
- 명확한 오류 메시지와 함께 빠른 실패
- 외부 데이터를 절대 신뢰하지 않음 (API 응답, 사용자 입력, 파일 내용)

## 코드 품질 체크리스트

작업 완료 전 확인:
- [ ] 코드가 읽기 쉽고 이름이 잘 지어져 있는가
- [ ] 함수가 작은가 (50줄 미만)
- [ ] 파일이 집중되어 있는가 (800줄 미만)
- [ ] 깊은 중첩이 없는가 (4레벨 초과)
- [ ] 적절한 오류 처리가 되어 있는가
- [ ] 하드코딩된 값이 없는가 (상수 또는 설정 사용)
- [ ] 변경이 없는가 (불변 패턴 사용)
