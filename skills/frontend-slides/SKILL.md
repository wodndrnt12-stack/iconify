# Frontend Slides

완전히 브라우저에서 실행되는 의존성 없는 애니메이션이 풍부한 HTML 프레젠테이션을 만듭니다.

[zarazhangrui](https://github.com/zarazhangrui)의 작업에서 선보인 시각적 탐색 방식에서 영감을 받았습니다.

## 활성화 시점

- 발표 덱, 피치 덱, 워크숍 덱, 또는 내부 프레젠테이션 만들기
- `.ppt` 또는 `.pptx` 슬라이드를 HTML 프레젠테이션으로 변환
- 기존 HTML 프레젠테이션의 레이아웃, 모션 또는 타이포그래피 개선
- 디자인 선호도를 아직 모르는 사용자와 프레젠테이션 스타일 탐색

## 필수 요건

1. **의존성 없음**: 기본적으로 CSS와 JS가 인라인된 하나의 독립된 HTML 파일로.
2. **뷰포트 맞춤 필수**: 모든 슬라이드는 내부 스크롤 없이 하나의 뷰포트에 맞아야 합니다.
3. **보여주기, 설명하지 않기**: 추상적인 스타일 설문 대신 시각적 미리보기 사용.
4. **독특한 디자인**: 일반적인 보라색-그라데이션, 흰 배경에 Inter 폰트, 템플릿 같은 덱 금지.
5. **프로덕션 품질**: 코드에 주석 달기, 접근 가능하고 반응형이며 성능 좋게 유지.

생성 전에 `STYLE_PRESETS.md`에서 뷰포트 안전 CSS 베이스, 밀도 제한, 프리셋 카탈로그, CSS 주의사항을 확인하세요.

## 워크플로우

### 1. 모드 감지

하나의 경로를 선택하세요:
- **새 프레젠테이션**: 사용자에게 주제, 노트, 또는 전체 초안이 있는 경우
- **PPT 변환**: 사용자에게 `.ppt` 또는 `.pptx`가 있는 경우
- **개선**: 사용자에게 이미 HTML 슬라이드가 있고 개선을 원하는 경우

### 2. 콘텐츠 파악

최소한만 질문하세요:
- 목적: 피치, 교육, 컨퍼런스 발표, 내부 업데이트
- 길이: 짧음 (5-10), 중간 (10-20), 긺 (20+)
- 콘텐츠 상태: 완성된 문구, 대략적인 메모, 주제만 있음

사용자에게 콘텐츠가 있으면 스타일 작업 전에 붙여넣도록 요청하세요.

### 3. 스타일 파악

기본적으로 시각적 탐색을 선택합니다.

사용자가 원하는 프리셋을 이미 알고 있다면 미리보기를 건너뛰고 바로 사용하세요.

그렇지 않으면:
1. 덱이 주어야 할 느낌을 물어보세요: 인상적, 활기차게, 집중, 영감.
2. `.ecc-design/slide-previews/`에 **3개의 단일 슬라이드 미리보기 파일**을 생성하세요.
3. 각 미리보기는 자체 포함되어야 하고, 타이포그래피/색상/모션을 명확히 보여줘야 하며, 슬라이드 콘텐츠가 약 100줄 이하여야 합니다.
4. 사용자에게 어떤 미리보기를 유지할지 또는 어떤 요소를 조합할지 물어보세요.

분위기를 스타일에 매핑할 때 `STYLE_PRESETS.md`의 프리셋 가이드를 사용하세요.

### 4. 프레젠테이션 구축

출력:
- `presentation.html`
- `[presentation-name].html`

덱에 추출된 또는 사용자 제공 이미지가 있는 경우에만 `assets/` 폴더를 사용하세요.

필수 구조:
- 시맨틱 슬라이드 섹션
- `STYLE_PRESETS.md`의 뷰포트 안전 CSS 베이스
- 테마 값을 위한 CSS 커스텀 프로퍼티
- 키보드, 휠, 터치 네비게이션을 위한 프레젠테이션 컨트롤러 클래스
- 리빌 애니메이션을 위한 Intersection Observer
- reduced-motion 지원

### 5. 뷰포트 맞춤 강제

이것을 하드 게이트로 처리하세요.

규칙:
- 모든 `.slide`는 `height: 100vh; height: 100dvh; overflow: hidden;`을 사용해야 합니다
- 모든 타이포그래피와 간격은 `clamp()`로 조정해야 합니다
- 콘텐츠가 맞지 않으면 여러 슬라이드로 분할
- 텍스트를 읽기 어려운 크기로 줄여서 오버플로우 해결 금지
- 슬라이드 내부에 스크롤바 절대 허용 금지

`STYLE_PRESETS.md`의 밀도 제한과 필수 CSS 블록을 사용하세요.

### 6. 검증

완성된 덱을 다음 크기에서 확인하세요:
- 1920x1080
- 1280x720
- 768x1024
- 375x667
- 667x375

브라우저 자동화를 사용할 수 있다면, 슬라이드 오버플로우 없음과 키보드 네비게이션 작동을 확인하는 데 사용하세요.

### 7. 전달

전달 시:
- 사용자가 유지를 원하지 않는 한 임시 미리보기 파일 삭제
- 유용한 경우 플랫폼에 맞는 오프너로 덱 열기
- 파일 경로, 사용된 프리셋, 슬라이드 수, 쉬운 테마 커스터마이징 포인트 요약

현재 OS에 맞는 올바른 오프너 사용:
- macOS: `open file.html`
- Linux: `xdg-open file.html`
- Windows: `start "" file.html`

## PPT / PPTX 변환

PowerPoint 변환의 경우:
1. `python-pptx`가 있는 `python3`를 사용하여 텍스트, 이미지, 노트를 추출하는 것을 선호합니다.
2. `python-pptx`를 사용할 수 없으면 설치 여부를 묻거나 수동/내보내기 기반 워크플로우로 대체합니다.
3. 슬라이드 순서, 발표자 노트, 추출된 에셋을 보존합니다.
4. 추출 후 새 프레젠테이션과 동일한 스타일 선택 워크플로우를 실행합니다.

변환은 크로스 플랫폼으로 유지하세요. Python이 작업을 할 수 있을 때 macOS 전용 도구에 의존하지 마세요.

## 구현 요구사항

### HTML / CSS

- 사용자가 멀티 파일 프로젝트를 명시적으로 원하지 않는 한 인라인 CSS와 JS 사용.
- 폰트는 Google Fonts 또는 Fontshare에서 가져올 수 있습니다.
- 대기 분위기의 배경, 강한 타이포그래피 계층, 명확한 시각적 방향을 선호하세요.
- 일러스트레이션 대신 추상적인 모양, 그라데이션, 격자, 노이즈, 기하학 사용.

### JavaScript

포함 항목:
- 키보드 네비게이션
- 터치 / 스와이프 네비게이션
- 마우스 휠 네비게이션
- 진행률 표시기 또는 슬라이드 인덱스
- 진입 시 리빌 애니메이션 트리거

### 접근성

- 시맨틱 구조 사용 (`main`, `section`, `nav`)
- 읽기 쉬운 대비 유지
- 키보드만으로 네비게이션 지원
- `prefers-reduced-motion` 준수

## 콘텐츠 밀도 제한

가독성이 유지되면서 사용자가 더 밀도 높은 슬라이드를 명시적으로 요청하지 않는 한 다음 최대값 사용:

| 슬라이드 유형 | 제한 |
|------------|-------|
| 제목 | 1 헤딩 + 1 부제목 + 선택적 태그라인 |
| 콘텐츠 | 1 헤딩 + 4-6 불릿 또는 짧은 단락 2개 |
| 기능 그리드 | 6개 카드 최대 |
| 코드 | 8-10줄 최대 |
| 인용 | 1 인용 + 출처 |
| 이미지 | 뷰포트로 제한된 1개 이미지 |

## 안티패턴

- 시각적 정체성 없는 일반적인 스타트업 그라데이션
- 의도적으로 편집적이지 않는 한 시스템 폰트 덱
- 긴 불릿 나열
- 스크롤이 필요한 코드 블록
- 짧은 화면에서 깨지는 고정 높이 콘텐츠 박스
- `-clamp(...)`와 같은 잘못 부정된 CSS 함수

## 관련 ECC 스킬

- `e2e-testing` — 최종 덱에 대한 자동화된 브라우저 검증이 필요할 때

## 결과물 체크리스트

- 프레젠테이션이 브라우저에서 로컬 파일로 실행됨
- 모든 슬라이드가 스크롤 없이 뷰포트에 맞음
- 스타일이 독특하고 의도적임
- 애니메이션이 의미 있고 시끄럽지 않음
- reduced motion이 준수됨
- 파일 경로와 커스터마이징 포인트가 전달 시 설명됨
