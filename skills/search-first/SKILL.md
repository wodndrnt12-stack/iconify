# /search-first — 코딩 전 먼저 조사하기

"구현하기 전에 기존 솔루션 검색" 워크플로우를 체계화합니다.

## 트리거

다음의 경우 이 스킬을 사용하세요:
- 기존 솔루션이 있을 가능성이 높은 새 기능 시작
- 의존성 또는 통합 추가
- 사용자가 "X 기능 추가"를 요청하고 코드를 작성하려는 경우
- 새 유틸리티, 헬퍼, 추상화를 만들기 전

## 워크플로우

```
┌─────────────────────────────────────────────┐
│  1. 필요 분석                               │
│     필요한 기능 정의                         │
│     언어/프레임워크 제약 파악                │
├─────────────────────────────────────────────┤
│  2. 병렬 검색 (researcher 에이전트)         │
│     ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│     │  npm /   │ │  MCP /   │ │ GitHub / │  │
│     │  PyPI    │ │  Skills  │ │  Web     │  │
│     └──────────┘ └──────────┘ └──────────┘  │
├─────────────────────────────────────────────┤
│  3. 평가                                    │
│     후보 채점 (기능, 유지보수, 커뮤니티,    │
│     문서, 라이선스, 의존성)                 │
├─────────────────────────────────────────────┤
│  4. 결정                                    │
│     ┌─────────┐  ┌──────────┐  ┌─────────┐  │
│     │  그대로  │  │  확장/   │  │  직접   │  │
│     │  채택   │  │  래핑    │  │  구축   │  │
│     └─────────┘  └──────────┘  └─────────┘  │
├─────────────────────────────────────────────┤
│  5. 구현                                    │
│     패키지 설치 / MCP 설정 /               │
│     최소한의 커스텀 코드 작성              │
└─────────────────────────────────────────────┘
```

## 의사결정 매트릭스

| 신호 | 액션 |
|--------|--------|
| 정확한 일치, 잘 유지됨, MIT/Apache | **채택** — 직접 설치하고 사용 |
| 부분 일치, 좋은 기반 | **확장** — 설치 + 얇은 래퍼 작성 |
| 여러 약한 일치 | **조합** — 2-3개의 작은 패키지 결합 |
| 적합한 것을 찾지 못함 | **구축** — 커스텀 작성, 단 조사 내용 반영 |

## 사용 방법

### 빠른 모드 (인라인)

유틸리티를 작성하거나 기능을 추가하기 전에 다음을 검토하세요:

0. 이것이 이미 레포에 존재하는가? → 관련 모듈/테스트를 먼저 `rg`로 검색
1. 이것이 일반적인 문제인가? → npm/PyPI 검색
2. 이를 위한 MCP가 있는가? → `~/.claude/settings.json` 확인 및 검색
3. 이를 위한 스킬이 있는가? → `~/.claude/skills/` 확인
4. GitHub에 구현/템플릿이 있는가? → 새 코드 작성 전에 유지 관리되는 OSS를 위해 GitHub 코드 검색

### 전체 모드 (에이전트)

비자명한 기능의 경우 researcher 에이전트를 실행하세요:

```
Task(subagent_type="general-purpose", prompt="
  다음을 위한 기존 도구 조사: [설명]
  언어/프레임워크: [언어]
  제약: [제약사항]

  검색: npm/PyPI, MCP 서버, Claude Code 스킬, GitHub
  반환: 권장사항이 있는 구조화된 비교
")
```

## 카테고리별 검색 단축키

### 개발 도구
- 린팅 → `eslint`, `ruff`, `textlint`, `markdownlint`
- 포맷팅 → `prettier`, `black`, `gofmt`
- 테스트 → `jest`, `pytest`, `go test`
- Pre-commit → `husky`, `lint-staged`, `pre-commit`

### AI/LLM 통합
- Claude SDK → 최신 문서를 위해 Context7
- 프롬프트 관리 → MCP 서버 확인
- 문서 처리 → `unstructured`, `pdfplumber`, `mammoth`

### 데이터 및 API
- HTTP 클라이언트 → `httpx` (Python), `ky`/`got` (Node)
- 유효성 검사 → `zod` (TS), `pydantic` (Python)
- 데이터베이스 → MCP 서버 먼저 확인

### 콘텐츠 및 퍼블리싱
- Markdown 처리 → `remark`, `unified`, `markdown-it`
- 이미지 최적화 → `sharp`, `imagemin`

## 통합 포인트

### planner 에이전트와 함께
planner는 1단계 (아키텍처 검토) 전에 researcher를 호출해야 합니다:
- Researcher가 사용 가능한 도구 파악
- Planner가 구현 계획에 통합
- 계획에서 "바퀴를 재발명"하는 것을 방지

### architect 에이전트와 함께
architect는 다음을 위해 researcher에 자문해야 합니다:
- 기술 스택 결정
- 통합 패턴 발견
- 기존 참조 아키텍처

### iterative-retrieval 스킬과 함께
점진적 발견을 위해 결합:
- 사이클 1: 광범위한 검색 (npm, PyPI, MCP)
- 사이클 2: 상위 후보를 상세히 평가
- 사이클 3: 프로젝트 제약과 호환성 테스트

## 예시

### 예시 1: "죽은 링크 확인 추가"
```
필요: 깨진 링크에 대해 markdown 파일 확인
검색: npm "markdown dead link checker"
발견: textlint-rule-no-dead-link (점수: 9/10)
액션: 채택 — npm install textlint-rule-no-dead-link
결과: 커스텀 코드 없음, 검증된 솔루션
```

### 예시 2: "HTTP 클라이언트 래퍼 추가"
```
필요: 재시도 및 타임아웃 처리가 있는 탄력적 HTTP 클라이언트
검색: npm "http client retry", PyPI "httpx retry"
발견: got (Node) with retry plugin, httpx (Python) with built-in retry
액션: 채택 — 재시도 설정과 함께 got/httpx 직접 사용
결과: 커스텀 코드 없음, 프로덕션 검증된 라이브러리
```

### 예시 3: "설정 파일 린터 추가"
```
필요: 스키마에 대해 프로젝트 설정 파일 유효성 검사
검색: npm "config linter schema", "json schema validator cli"
발견: ajv-cli (점수: 8/10)
액션: 채택 + 확장 — ajv-cli 설치, 프로젝트별 스키마 작성
결과: 1개 패키지 + 1개 스키마 파일, 커스텀 유효성 검사 로직 없음
```

## 안티패턴

- **코드로 바로 뛰어들기**: 기존 것이 있는지 확인하지 않고 유틸리티 작성
- **MCP 무시**: MCP 서버가 이미 기능을 제공하는지 확인하지 않음
- **과도한 커스터마이징**: 라이브러리를 혜택을 잃을 만큼 심하게 래핑
- **의존성 비대**: 하나의 작은 기능을 위해 거대한 패키지 설치
