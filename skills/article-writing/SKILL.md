# 기사 작성

일반적인 AI 출력이 아닌 실제 사람이나 브랜드처럼 들리는 장문 콘텐츠를 작성합니다.

## 활성화 시점

- 블로그 게시물, 에세이, 런칭 포스트, 가이드, 튜토리얼, 뉴스레터 초안 작성 시
- 메모, 녹취록, 리서치를 완성된 기사로 전환 시
- 예시에서 기존 창업자, 운영자, 브랜드 목소리 매칭 시
- 이미 작성된 장문 카피의 구조, 흐름, 근거 강화 시

## 핵심 규칙

1. 구체적인 것부터 시작: 예시, 출력물, 일화, 숫자, 스크린샷 설명, 코드 블록.
2. 예시 후에 설명, 그 이전이 아님.
3. 군더더기 없이 짧고 직접적인 문장 선호.
4. 출처가 있는 구체적인 숫자 사용.
5. 전기적 사실, 회사 지표, 고객 증거를 절대 임의로 만들지 말 것.

## 목소리 포착 워크플로우

사용자가 특정 목소리를 원하면 다음 중 하나 이상을 수집:
- 게시된 기사
- 뉴스레터
- X / LinkedIn 게시물
- 문서 또는 메모
- 간단한 스타일 가이드

그런 다음 추출:
- 문장 길이와 리듬
- 형식적, 대화적, 날카로운 목소리 여부
- 괄호, 목록, 단편, 질문 등 선호하는 수사법
- 유머, 의견, 반론적 프레이밍에 대한 허용도
- 헤더, 불릿, 코드 블록, 인용구 등 포맷 습관

목소리 참조가 없으면 직접적이고 운영자 스타일의 목소리를 기본값으로: 구체적이고 실용적이며 과장 없음.

## 금지 패턴

다음을 삭제하고 다시 작성:
- "오늘날 빠르게 변화하는 환경에서"와 같은 일반적인 도입부
- "게다가", "더욱이"와 같은 채워넣기 전환
- "게임 체인저", "최첨단", "혁명적"과 같은 과장 표현
- 근거 없는 막연한 주장
- 제공된 맥락으로 뒷받침되지 않는 전기나 신뢰성 주장

## 작성 프로세스

1. 독자와 목적 명확화.
2. 섹션당 하나의 목적이 있는 골격 아웃라인 작성.
3. 각 섹션을 증거, 예시, 장면으로 시작.
4. 다음 문장이 자리를 얻을 때만 확장.
5. 틀에 박히거나 자화자찬처럼 들리는 것 제거.

## 구조 지침

### 기술 가이드
- 독자가 얻을 것으로 시작
- 모든 주요 섹션에 코드 또는 터미널 예시 사용
- 부드러운 요약이 아닌 구체적인 핵심 사항으로 마무리

### 에세이 / 의견 글
- 긴장감, 모순, 날카로운 관찰로 시작
- 섹션당 하나의 논쟁 실마리
- 의견을 정당화하는 예시 사용

### 뉴스레터
- 첫 화면을 강하게 유지
- 일기 내용이 아닌 통찰과 업데이트 혼합
- 명확한 섹션 레이블과 쉽게 훑어볼 수 있는 구조

## 품질 관문

전달 전:
- 제공된 출처에 대해 사실 주장 확인
- 채워넣기 및 기업 언어 제거
- 목소리가 제공된 예시와 일치하는지 확인
- 모든 섹션이 새로운 정보를 추가하는지 확인
- 의도한 플랫폼에 맞는 포맷 확인
