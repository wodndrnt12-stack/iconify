# 문서 업데이트

코드베이스와 문서를 동기화하고, 신뢰할 수 있는 소스 파일에서 생성합니다.

## 1단계: 신뢰할 수 있는 소스 식별

| 소스 | 생성 대상 |
|------|-----------|
| `package.json` 스크립트 | 사용 가능한 명령어 참조 |
| `.env.example` | 환경 변수 문서 |
| `openapi.yaml` / 라우트 파일 | API 엔드포인트 참조 |
| 소스 코드 익스포트 | 공개 API 문서 |
| `Dockerfile` / `docker-compose.yml` | 인프라 설정 문서 |

## 2단계: 스크립트 참조 생성

1. `package.json` (또는 `Makefile`, `Cargo.toml`, `pyproject.toml`) 읽기
2. 설명과 함께 모든 스크립트/명령어 추출
3. 참조 표 생성:

```markdown
| 명령어 | 설명 |
|--------|------|
| `npm run dev` | 핫 리로드로 개발 서버 시작 |
| `npm run build` | 타입 검사와 함께 프로덕션 빌드 |
| `npm test` | 커버리지와 함께 테스트 스위트 실행 |
```

## 3단계: 환경 문서 생성

1. `.env.example` (또는 `.env.template`, `.env.sample`) 읽기
2. 목적과 함께 모든 변수 추출
3. 필수 vs 선택으로 분류
4. 예상 형식과 유효한 값 문서화

```markdown
| 변수 | 필수 | 설명 | 예시 |
|------|------|------|------|
| `DATABASE_URL` | 예 | PostgreSQL 연결 문자열 | `postgres://user:pass@host:5432/db` |
| `LOG_LEVEL` | 아니오 | 로깅 상세도 (기본값: info) | `debug`, `info`, `warn`, `error` |
```

## 4단계: 기여 가이드 업데이트

다음을 포함하여 `docs/CONTRIBUTING.md` 생성 또는 업데이트:
- 개발 환경 설정 (사전 요구사항, 설치 단계)
- 사용 가능한 스크립트와 목적
- 테스트 절차 (실행 방법, 새 테스트 작성 방법)
- 코드 스타일 적용 (린터, 포매터, pre-commit 훅)
- PR 제출 체크리스트

## 5단계: 런북 업데이트

다음을 포함하여 `docs/RUNBOOK.md` 생성 또는 업데이트:
- 배포 절차 (단계별)
- 헬스 체크 엔드포인트 및 모니터링
- 일반적인 문제와 해결 방법
- 롤백 절차
- 알림 및 에스컬레이션 경로

## 6단계: 오래됨 확인

1. 90일 이상 수정되지 않은 문서 파일 찾기
2. 최근 소스 코드 변경사항과 교차 참조
3. 수동 검토가 필요한 잠재적으로 오래된 문서 표시

## 7단계: 요약 표시

```
문서 업데이트
──────────────────────────────
업데이트: docs/CONTRIBUTING.md (스크립트 표)
업데이트: docs/ENV.md (새 변수 3개)
표시됨:   docs/DEPLOY.md (142일 오래됨)
건너뜀:   docs/API.md (변경사항 없음)
──────────────────────────────
```

## 규칙

- **단일 신뢰 소스**: 항상 코드에서 생성, 생성된 섹션 수동 편집 금지
- **수동 섹션 보존**: 생성된 섹션만 업데이트; 수작업으로 작성된 내용 그대로 유지
- **생성된 콘텐츠 표시**: 생성된 섹션 주변에 `<!-- AUTO-GENERATED -->` 마커 사용
- **무분별한 문서 생성 금지**: 명령어가 명시적으로 요청한 경우에만 새 문서 파일 생성
