# Plan 명령어

이 명령어는 코드 작성 전 포괄적인 구현 계획을 수립하기 위해 **planner** 에이전트를 호출합니다.

## 이 명령어가 하는 일

1. **요구사항 재명시** — 무엇을 만들어야 하는지 명확히 정리
2. **리스크 파악** — 잠재적 문제와 블로커 발굴
3. **단계별 계획 수립** — 구현을 단계로 분해
4. **확인 대기** — 진행 전 반드시 사용자 승인 수령

## 사용 시기

다음 경우에 `/plan` 사용:
- 새 기능 시작 시
- 중요한 아키텍처 변경 시
- 복잡한 리팩토링 작업 시
- 여러 파일/컴포넌트에 영향을 미치는 경우
- 요구사항이 불명확하거나 모호한 경우

## 동작 방식

planner 에이전트는 다음을 수행합니다:

1. **요청 분석** 및 명확한 표현으로 요구사항 재명시
2. **단계별 분해** — 구체적이고 실행 가능한 단계로
3. **의존성 파악** — 컴포넌트 간 의존관계
4. **리스크 평가** 및 잠재적 블로커
5. **복잡도 추정** (높음/중간/낮음)
6. **계획 제시** 및 명시적 확인 대기

## 사용 예시

```
사용자: /plan 마켓 정산 시 실시간 알림 추가가 필요합니다

에이전트 (planner):
# 구현 계획: 실시간 마켓 정산 알림

## 요구사항 재명시
- 사용자가 주시하는 마켓 정산 시 알림 발송
- 다중 알림 채널 지원 (인앱, 이메일, 웹훅)
- 알림 안정적 전달 보장
- 마켓 결과 및 사용자 포지션 결과 포함

## 구현 단계

### 1단계: 데이터베이스 스키마
- notifications 테이블 추가 (컬럼: id, user_id, market_id, type, status, created_at)
- 채널 선호도를 위한 user_notification_preferences 테이블 추가
- 성능을 위해 user_id 및 market_id에 인덱스 생성

### 2단계: 알림 서비스
- lib/notifications.ts에 알림 서비스 생성
- BullMQ/Redis를 사용한 알림 큐 구현
- 전달 실패 시 재시도 로직 추가
- 알림 템플릿 생성

### 3단계: 통합 지점
- 마켓 정산 로직 연결 (상태가 "resolved"로 변경될 때)
- 해당 마켓에 포지션이 있는 모든 사용자 조회
- 각 사용자에게 알림 큐에 추가

### 4단계: 프론트엔드 컴포넌트
- 헤더에 NotificationBell 컴포넌트 생성
- NotificationList 모달 추가
- Supabase 구독을 통한 실시간 업데이트 구현
- 알림 설정 페이지 추가

## 의존성
- Redis (큐용)
- 이메일 서비스 (SendGrid/Resend)
- Supabase 실시간 구독

## 리스크
- 높음: 이메일 전달성 (SPF/DKIM 필요)
- 중간: 마켓당 1,000명 이상 사용자 시 성능
- 중간: 마켓 정산이 잦을 경우 알림 스팸
- 낮음: 실시간 구독 오버헤드

## 예상 복잡도: 중간
- 백엔드: 4~6시간
- 프론트엔드: 3~4시간
- 테스트: 2~3시간
- 합계: 9~13시간

**확인 대기 중**: 이 계획대로 진행할까요? (yes/no/modify)
```

## 중요 참고사항

**핵심**: planner 에이전트는 "yes" 또는 "proceed" 등 명시적인 확인 응답을 받기 전까지 **코드를 작성하지 않습니다**.

변경을 원하는 경우 다음과 같이 응답하세요:
- "modify: [변경 사항]"
- "different approach: [대안]"
- "skip phase 2 and do phase 3 first"

## 다른 명령어와의 통합

기획 후:
- 테스트 주도 개발로 구현하려면 `/tdd` 사용
- 빌드 오류 발생 시 `/build-fix` 사용
- 완성된 구현 검토 시 `/code-review` 사용
